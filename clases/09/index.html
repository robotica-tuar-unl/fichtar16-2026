Gazebo

  • Importancia de los entornos de simulación
  • Gratuito y open-source
  • Mantenido por Open Robotics (ROS)
gazebosim.org
  • Probar en hardware real muchas veces es caro y además consume mucho esfuerzo y tiempo
  • Las acciones de ciertos robots pueden contener peligro y “deployar” código no probado puede suponer un alto riesgo
  • A veces el hardware real puede introducir una serie de problemas que no permiten el correcto desarrollo de software (relacionado con aislar variables)
  • Por estas razones es apropiado desarrollar entornos de simulación precisos. Gazebo es uno de ellos, gratuito y open-source.
  • El proyecto es mantenido por el grupo “Open Robotics” que se encuentra detrás de ROS también, sin embargo son dos proyectos separados.

Documentación: gazebosim.org/docs/harmonic

Gazebo

Entorno y robot real ➡️ Entorno virtual y robot simulado

  • Lectura de sensores simulados
  • Comandar actuadores virtuales
  • Obtener feedback de los comandos

Con Gazebo se crea un entorno o mundo virtual, y se carga la versión simulada del robot.

Los sensores simulados pueden detectar el entorno virtual y publicar la información a los mismos topics de ROS como un sensor real lo haría, permitiendo probar algoritmos de forma fácil.

Luego se pueden aplicar comandos a los actuadores simulados en el robot para así probar la respuesta del sistema.

Sistema con joint_state_publisher_gui

Se utilizaba joint_state_publisher_gui para crear datos fakes en el topic /joint_states para que el robot_state_publisher publique las transformaciones

Sistema con Gazebo

Con Gazebo se busca representar el mundo real, un robot físico con el que interactuar de varias maneras. Por ejemplo, que se mueva basado en una entrada de control, leer la información de los sensores, y obtener un feedback de los actuadores.

Ejcutar Gazebo con ROS2

Se utiliza el launch del paquete ros_gz_sim

    $ ros2 launch ros_gz_sim gz_sim.launch.py gz_args:=empty.sdf

Para agregarlo a un archivo launch

    # .. 
    IncludeLaunchDescription(
        PythonLaunchDescriptionSource(

            ]),
        ),
        launch_arguments={
            'gz_args': '-r empty.sdf',
            'on_exit_shutdown': 'True'
        }.items(),
    ),
    # ..

Recordar actualizar las dependencias de ejecución

    <exec_depend>ros_gz</exec_depend>

SDF: Simulation Description Format

  • Gazebo utiliza archivos SDF (≠ URDF)
  • Además de modelos, permite describir escenarios (worlds)

Se puede convertir URDF a SDF, pero se necesitan ajustes

No es necesario escribir un URDF y un SDF para el robot, Gazebo convierte el primero en el segundo, siempre y cuando se realicen ciertas modificaciones o agregados para que pueda trabajar correctamente con Gazebo.

Esto permite combinar nuestro robot con variedad de modelos y entornos creados por la comunidad (online)

Gazebo y URDF

  • Para agregar elementos de Gazebo al URDF:

<gazebo> .. </gazebo>

  • Para hacer referencia a elementos del URDF:

<gazebo name="[nombre_link|joint|etc]">

</gazebo>

Para agregar elementos específicos de Gazebo se utilizan tags <gazebo></gazebo>. Esto permite capturar el contenido para generar los archivos SDF e ignorar el resto de contenido.

Gazebo y URDF

Requisitos:

  • Nombramiento:
    cada link y cada junta con nombre único
  • Propiedades inerciales:
    cada link debe tener su tag inertial
  • Materiales y colores:
    Gazebo no utiliza los materiales definidos en el URDF
  • Nombres o nombramiento: Para evitar conflictos se recomienda nombrar cada junta y cada link con un nombre único (ej utilizando _joint y _link respectivamente)
  • Combinación de links: Gazebo toma los links unidos por juntas fijas y los combina en un solo SDF link, no sería un problema pero puede quedar confuso a veces
  • Propiedades inerciales: Gazebo requiere que cada link posea su tag inertial asociado
  • World links: En el caso de que el robot esté fijo al suelo, el primer link debe ser nombrado world, y Gazebo mantendrá fijo ese link más allá del de la física. Para un robot móvil el primer link NO debe llamarse world
  • Materiales y colores: Gazebo no utiliza los materiales definidos en el URDF. Es necesario utilizar el tag gazebo para cada link y especificar el material apropiado allí

Gazebo y URDF: <collision>

Deben definirse los elementos de colisión para cada link

<link name="[nombre_link]">
    ..
    <collision>
        <origin xyz="[origen]" rpy="[orientacion]"/>
        <geometry>
            <!-- Prisma -->
            <box size="[largo] [ancho] [alto]"/>
            <!-- Cilindro -->
             <cylinder radius="[radio]" length="[ancho]" />
            <!-- Esfera -->
            <sphere radius="[radio]" />
        </geometry>
    </collision>
    ..
</link>
  • El tag de colisión es al mismo nivel que el tag <visual>
  • El elemento de colisión define la geometría igual que el visual
  • Se puede especificar un origen específico

Gazebo y URDF: <inertial>

Deben definirse las propiedades físicas para cada link

<link name="[nombre_link]">
    ..
    <inertial>
        <origin xyz="[traslacion]" rpy="[rotacion]"/>
        <mass value="[masa_kg]"/>
        <inertia ixx="[I_xx]" ixy="[I_xy]" ixz="[I_xz]"
                              iyy="[I_yy]" iyz="[I_yz]"
                                           izz="[I_zz]"/>
    </inertial>
    ..
</link>
  • La masa se define en kilogramo
  • La matriz 3x3 rotacional de inercia se especifica con el tag inertia
  • Al ser simétrica se especifican los 6 elementos por separado
  • La inercia depende de la distribución de la masa del objeto
  • Mediante el tag origin se especifica el centro de gravedad

Tensores de inercia

inertial_macros.xacro

Archivo XACRO para el cálculo de las inercias según la geometría

  • Prisma
    <xacro:inertial_box mass="[masa_kg]" x="[largo]" y="[ancho]" z="[alto]">
        <origin xyz="[traslacion]" rpy="[rotacion]"/>
    </xacro:inertial_box>
  • Cilindro
    <xacro:inertial_cylinder mass="[masa_kg]" radius="[radio]" length="[ancho]">
        <origin xyz="[traslacion]" rpy="[rotacion]"/>
    </xacro:inertial_cylinder>
  • Inercia despreciable
    <dummy_inertial />

Control de robots reales y simulados

Sistemas robóticos

  1. Sense: Entrada (usuario / entorno)

  2. Think: Calcular o procesar

  3. Act: Controlar un actuador


Existen enorme diversidad de actuadores y métodos de control pero a la vez son siempre similares

  • Por lo general, el elemento principal de un robot es tomar algún tipo de entrada (de un operador o del entorno), hacer algo “inteligente” y según el resultado controlar un actuador

  • Existen muchos tipos diferentes de actuadores e interfaces, y muchas metodologías de control distintas para resolver problemas

  • Pero dentro de. enorme universo de opciones, en realidad hay muchas similitudes

  • Si no se cuenta con algún tipo de sistema estandarizado y siempre se programa un controlador y una interfaz personalizados para cada aplicación terminamos perdiendo el tiempo reescribiendo cosas y resolviendo problemas que ya se han resuelto cientos de veces

  • Podría pensarse que este problema se presta bien a los topics de ROS: mensajes estándar, escribir nodos para los controladores y drivers, ¡y listo!

    • Un problema con este enfoque es que queremos que algo así se ejecute a máxima velocidad
    • no debería haber ningún retraso entre el controlador y el driver

  • Solución ros2_control

Paquete ros2_control

Controller manager

  • Es el centro del sistema
  • Se encarga de conectar los algoritmos de control con los elementos de “hardware”
  • Utiliza un sistema de plugins que carga los componentes según la ejecución


Paquete ros2_control

Hardware interfaces

  • Capa entre el hardware y ros2_control
  • 2 robots que se controlan igual pueden tener diferencias de implementación:
    • Comunicación de los motores
    • Variable que controlan
    • Único o múltiples puertos
    • etc.


  • Actúa como capa de abstracción entre el hardware real (o simulado) y ros2_control

Paquete ros2_control

Hardware interfaces

  • Capa entre el hardware y ros2_control
  • 2 robots que se controlan igual pueden tener diferencias de implementación
  • El hardware interface aísla estos detalles para que el usuario solo utilice 2 interfaces:
    • state interfaces:
      estado / sensor (entrada)
    • command interfaces:
      actuador (salida)


  • Mediante la capa de hardware interface, el usuario solo tiene que entender como se representa (no los detalles de implementación en sí)
  • La representación se simplifica en 2 tipos de interfaces:
    • Interfaces de comando: cosas que se pueden controlar
    • Interfaces de estado: cosas que solo se pueden monitorear
  • Lo importante es que se ocultan los detalles del robot y solo se trabaja con interfaces

Paquete ros2_control

Hardware interfaces

  • Un robot puede tener múltiples hardware interface
  • Cada uno a su vez distinta cantidad de interfaces


  • Ejemplo de un brazo robótico sobre una base móvil cada uno con su sistema de control de motores separado o cada motor independiente

  • Ejemplo de un robot como el del entregable (diff drive)

    • Actuadores: motores -> control de velocidad
    • Cantidad 2 -> Hardware interface tendrá 2 interfaces de comando
    • Sensores: encoders por cada motor
    • Variables a medir con enconders -> Velocidad y posición (2)
    • Cantidad de interfaces de estado -> 2 motores x 2 variables = 4

  • En caso de medir torque o corriente se podrían agregar

Paquete ros2_control

Resource manager

  • Permite que los controladores accedan a la lista de las interfaces
  • El manejador de recursos y los controladores conocen las interfaces a través del URDF:

<ros2_control>..</ros2_control>


  • Para lidiar con el problema de los múltiples hardware interfaces y múltiples interfaces, ros2_control utiliza un manejador de recursos

Paquete ros2_control

Controllers

  • Puerta de comunicación con el resto del ecosistema de ROS2:
    • Suscriben a topics para recibir información
    • Procesan esa información a través de un algoritmo
    • Envian comandos a las interfaces de control y reciben el estado
    • Publican en topics información de estado


  • Los controladores son el medio por el cual el resto del ecosistema ROS interactúa con ros2_control
  • Por un lado, escuchan un topic de ROS para recibir información de control (como la posición de las articulaciones o la velocidad del cuerpo móvil)
  • Procesan esta información y utilizan un algoritmo para determinar la velocidad y posición adecuadas de los actuadores
  • Envían a las interfaces de hardware los controles correspondientes
  • También pueden publicar información en topics como retroalimentación de comandos o información de estado del hardware
  • Un solo controlador puede transmitir información en ambas direcciones
  • Es común encontrar controladores que en realidad no “controlan” nada, sino que simplemente publican datos de sensores o retroalimentación

Paquete ros2_control

Controllers

  • SI BIEN EL HARDWARE ES ESPECÍFICO LOS CONTROLADORES ESTÁN AISLADOS Y YA EXISTEN MÚLTIPLES PROGRAMADOS

  • PAQUETE: ros2_controllers


Paquete ros2_control

Resumiendo

  • La tarea del controller manager es tomar los controladores solicitados y asociarlos a las interfaces de estado y de comando adecuadas
  • Para configurar los controladores se utiliza un archivo YAML

  • Importante:

    • Una interfaz de control solo puede tener asociado un controlador
    • Una interfaz de estado sí puede asociarse a múltiples controladores


Paquete ros2_control

Controller manager

  • Para la ejecución se utiliza el nodo ros2_control_node dentro del paquete controller_manager:
    $ ros2 run controller_manager ros2_control_node [archivo_parametros.yaml]


  • Para robots en simulación el paquete gazebo_ros2_control ejecuta su propio controller manager

Se necesitan los detalles del hardware interface en el URDF y de los controllers en el YAML

Paquete ros2_control

Controller manager

  • Una vez que se encuentra en ejecución se necesita interactuar para chequear el hardware y lanzar los controllers
  • Existen 3 opciones:
    1. A través de servicios: ros2 service call ..
    2. Con comandos específicos: ros2 control ..
    3. Con nodos ya programados (que llaman a los servicios)

Modificaciones al URDF

  • Añadir el plugin de gz_ros2_control a Gazebo:
<gazebo>
    <plugin filename="gz_ros2_control-system"
            name="gz_ros2_control::GazeboSimROS2ControlPlugin" />
</gazebo>
  • Definir un hardware simulado con la etiqueta ros2_control:
<ros2_control name="GazeboSystem" type="system">
    <hardware>
        <plugin>gz_ros2_control/GazeboSimSystem</plugin>
    </hardware>
    <!-- Definición de las interfaces -->
</ros2_control>


Se recomienda crear un archivo aparte ros2_control.xacro e incorporarlo al robot description

Modificaciones al URDF

Configurar el hardware interface (tag ros2_control)

  • Las interfaces van a estar asociadas a juntas del URDF
  • Para definir las interfaces según el tipo
    • Interfaces de estado: <state_interface ../>
      <joint name="[nombre_junta]">
      <state_interface name="[position|velocity|effort]" />
  </joint>
    • Interfaces de comando: <command_interface .. />
      <joint name="[nombre_junta]">
      <command_interface name="[velocity|position|effort]">
          <!-- Parámetros opcionales -->
          <param name="min">[limite_min]</param>
          <param name="max">[limite_max]</param>
          <param name="initial_value">[valor_incial]</param>
      </command_interface>
  </joint>

Modificaciones al URDF

Ejemplo control de velocidad y sensado de posición y velocidad

ros2_control.xacro
<ros2_control name="GazeboSystem" type="system">
    <hardware>
        <plugin>gz_ros2_control/GazeboSimSystem</plugin>
    </hardware>
    <joint name="[nombre_junta]">
        <command_interface name="velocity">
            <!-- Límite de 10 rpm -->
            <param name="min">-1.047198</param>
            <param name="max">1.047198</param>
        </command_interface>
        <state_interface name="position" />
        <state_interface name="velocity" />
    </joint>
    <!-- .. -->
</ros2_control>

Modificaciones al URDF

Configurar los controladores (tag gazebo)

  • El plugin gz_ros2_control::GazeboSimROS2ControlPlugin realiza las siguientes acciones:
    • Carga los elementos necesarios para que Gazebo se comunique con el hardware interface
    • Ejecuta un controller manager
    • Busca el URDF provisto por el robot_state_publisher
  • Para el controller manager se deben asignar parámetros mediante un archivo YAML
<gazebo>
    <plugin name="gz_ros2_control::GazeboSimROS2ControlPlugin" filename="libgz_ros2_control-system.so">
        <parameters>$(find [nombre_paquete])/config/[nombre_archivo].yaml</parameters>
    </plugin>
</gazebo>

Parámetros del controller manager

Para definir los controladores se deben listar con:

  • Nombre
  • Tipo (según lista de controladores disponibles)
controllers.yaml
controller_manager:
  ros__parameters:
    update_rate: 30         # Velocidad de actualización de los controladores
    use_sim_time: true      # Utilizar la simulación de Gazebo

    <nombre_controlador>
        type: <tipo_controlador>
    
    <nombre_controlador2>
        type: <tipo_controlador>
    # ..

Los utilizados serán velocity_controllers y joint_state_broadcaster

Parámetros del controller manager

Controlador: joint_state_broadcaster

  • Tipo: joint_state_broadcaster/JointStateBroadcaster
  • Función: Leer todas las interfaces de estado y reportarlas en el topic /joint_states
controllers.yaml
<nombre_controlador>:
  ros__parameters:

    # Listado de juntas a publicar (si no se define se publican todas)
    joints: ["<nombre_junta1>", "<nombre_junta2>", "<..>" ]
    
    # Marco de referencia (por defecto es 'base_link')
    frame_id: <nombre_marco_referencia>


El orden en que se publican las juntas puede ser establecido con el parámetro joints

Documentación de comandos

Cargar controladores

Desde la linea de comando utilizando ros2 control:

    $ ros2 control load_controller --set-state active <nombre_controlador>

Desde archivo launch ejecutando el comando:

    ExecuteProcess(
        cmd = ['ros2', 'control', 'load_controller', '--set-state', 'active',
               '<nombre_controlador>'],
        output = 'screen'
    )

Desde archivo launch utilizando el script:

    Node(
        package = "controller_manager",
        executable = "spawner",
        arguments = [ "<nombre_controlador>" ],
    )
