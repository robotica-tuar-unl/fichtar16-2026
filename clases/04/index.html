Entorno y espacios de trabajo

Dos niveles:

  • overlay   ➡️   Nuestros paquetes
  • underlay   ➡️   Paquetes instalados


Workspace de desarrollo
source install/local_setup.bash
Paquete de ROS instalados
source /opt/ros/jazzy/setup.bash

  • ROS2 se basa en combiar espacios de trabajo (workspaces) usando el entorno de la terminal

  • Workspace: es la ubicación en el sistema donde se desarrolla con ROS2

  • El workspace núcleo se denomina underlay (subyacente) y el local se denomina overlay (superposición)

  • Es común que existan múltiples workspace activos al mismo tiempo

  • Este esquema facilita el desarrollo ante diferentes versiones de ROS o diferentes conjuntos de paquetes

  • Permite instalar distintas distribuciones de ROS en la misma PC y cambiar rapidamente entre ellas

  • Para lograr esto se aprovisionan (sourcing) archivos de setup cada vez que se abre una terminal

  • Al hacer source del overlay, trae las dependencias del underlay, pero sobre-escribe paquetes de igual nombre.

Workspace

Workspace: Estructura de carpetas que contiene a los paquetes

  • src: Paquetes y su código fuente
  • build: Archivos de compilación
  • install: Paquetes “instalados”
  • log: Información de log

Estructura

📂 ros_ws
├── 📁 src
├── 📁 build      ⚠️
├── 📁 install    ⚠️
└── 📁 log        ⚠️


Es común nombrar la carpeta raíz con un “_ws” al final

  • Antes de programar la primera linea de código se necesita un “lugar” donde escribir -> Workspace
  • Por lo general se nombra con ros2_ws
  • Al “construir” el workspace se crean 4 carpetas específicas
    • src: código fuente (Python o C++). Único lugar donde se modifican archivos
    • build: Donde se realiza la compilación o traducción a ejecutables
    • install: Una vez construidos, los ejecutables se mueven acá
    • log: Información acerca de fallos

Paquetes de ROS

Unidad organizativa del código de ROS2

  • Elemento base y más importante
  • Permite instalar código propio y compartirlo
  • Al liberar el paquete, permite a otros compilarlo y utilizarlo de forma fácil
  • Es el elemento más básico en la infraestructura de ROS, es importante saber crearlos de forma correcta y consistente
  • Todo lo que ofrece ROS gira en torno al uso efectivo de paquetes
  • Paquete: una unidad organizativa (contenedor o marco del código) para el código de ROS2
  • En caso de querer instalar o compartir código con otros se necesitará organizarlo en paquetes
  • Utilizando paquetes se puede liberar el trabajo, permitiendo a otros compilarlo y utilizarlo de forma fácil

    • /<nombre_paquete> -> Carpeta con el mismo nombre del paquete junto con el archivo __init__.py utilizado por las herramientas de ROS2

Paquetes de ROS

📄 package.xml

  • Propiedades e información del paquete en formato XML
  • Necesario:
    • <name>
    • <description>
    • <version>
    • <maintainer>
    • <license>

  • Más importante: dependencias

<?xml version="1.0"?>
<?xml-model
    href="http://download.ros.org/schema/package_format3.xsd"
    schematypens="http://www.w3.org/2001/XMLSchema"?>

<package format="3">
    <name>nombre_paquete</name>
    <version>0.0.0</version>
    <description>Descripción del paquete</description>
    <maintainer email="user@todo.todo">user</maintainer>
    <license>Licencia</license>

    <url type="website">https://github.com/...</url>
    <author email="user@todo.todo">user</author>

    <depend>nombre_paquete_dependencia</depend>

    <export>
        <build_type>ament_python</build_type>
    </export>
</package>
  • Nexo en común: todos los paquetes de ROS tienen este archivo
  • Propiedades e información del paquete: Nombre, versión, descripción y lo más importante, las dependencias
  • Dependencias: otros paquetes (por lo general de ROS) que el código necesita para funcionar
  • Será necesario completar: descripción, licencia y el “mainteiner”
  • Otros datos: autores, URL (repositorio),

Breve descripción de los archivos con codificación eXtensible Markup Language: - Lenguaje de marcado -> una forma de organizar la información utilizando etiquetas (para definir elementos) y atributos (para información adicional), de forma jerárquica. - Es flexible y puede personalizarse -> Útil para intercambiar información - Fácil de entender para humanos

Paquetes de ROS

📄 setup.py

  • Repite: nombre, descripción, versión, “maintainer”, y licencia
  • Debe coincidir la información con el package.xml
  • Definición de ejecutables para el comando ros2 run
from setuptools import setup

package_name = 'nombre_paquete'

setup(
  name=package_name,
  version='0.0.0',
  # Packages to export
  packages=[package_name],
  # Files we want to install, specifically launch files
  data_files=[
      # Install marker file in the package index
      ('share/ament_index/resource_index/packages',
          ['resource/' + package_name]),
      # Include our package.xml file
      ('share/' + package_name, ['package.xml']),
    ],
  # This is important as well
  install_requires=['setuptools'],
  zip_safe=True,
  maintainer='TODO',
  maintainer_email='TODO',
  description='TODO: Package description',
  license='TODO: License declaration',
  tests_require=['pytest'],
  entry_points={
      'console_scripts': [
              'nombre_nodo = nombre_paquete.nombre_nodo:main'
      ],
    },
)
  • Repite nombre, version, descripción, maintainer y licencia (es necesario que sean iguales en los dos archivos)

REP144 - Nombramiento de paquetes

Reglas para nombrar paquetes


  • Solo contienen caracteres alfanuméricos en minúscula
  • Solo se puede utilizar de separador el guión bajo “_”
  • No puede iniciar con “_” o contener doble guión “__”
  • Al menos 2 caracteres de longitud (se recomiendan muchos más)

  • Propone reglas para nombrar paquetes de ROS de forma apropiada (algunas son obligatorias, otras son recomendaciones)

  • El nombre de paquete se utiliza como namespace

  • Obligatoriamente:

    • Solo consiste en caracteres alfanumericos en minúscula (hay OS que no distinguen mayúscula y minúscula)
    • Separador ’_’ (no puede iniciar ni tener varios seguidos) -> los nombres deben ser soportados por todos los lenguajes
    • Al menos 2 caracteres de longitud (se recomienda que sea mucho más largo)

  • Reglas más generales:

    • Deben ser lo más específicos posible (para entender lo que hacen rapidamente)
    • Evitar nombres genéricos
    • Evitar utilizar “ros” en el nombre (para paquetes que no son base)
    • Evitar utilizar nombres ya ocupados (ver el index de ROS)

rclpy: Función de callback

Función que se genera y se envía para que sea ejecutada al ocurrir un evento


Permite al programa:

  • No detenerse a esperar el evento
  • Ejecutar la acción solo cuando es necesario
  • Continuar la ejecución de otras tareas

Acción que se ejecuta automáticamente al ocurrir un evento Porción de código que es pasada como argumento, a otra parte de código, que se espera sea ejecutado (call-back) en un momento conveniente.
