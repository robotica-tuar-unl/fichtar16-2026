Argumentos en ROS2

Conjunto de argumentos que permiten reconfigurar los nodos (solo al inicio)

    $ ros2 run <nombre_paquete> <nombre_ejecutable> --ros-args ...


Remapping: -r o --remap

  • Cambio de nombre de nodo
    $ ros2 run ... --ros-args -r __node:=<nuevo_nombre_nodo>
  • Cambio de namespace
    $ ros2 run ... --ros-args -r __ns:=/<nombre_namespace>
  • Cambio de nombre de topic
    $ ros2 run ... --ros-args -r <topic>:=<nuevo_nombre_topic>
  • Todos los nodos ROS aceptan un conjunto de argumentos que permiten reconfigurar diversas propiedades
  • De forma nativa (no hay que programar nada adicional)
  • Se ubican luego de la opción --ros-args

Argumentos en ROS2

Propiedades que ROS permite reconfigurar de forma nativa

Nivel de log: --log-level

    $ ros2 run ... --ros-args --log-level <info|debug|warn|error>


Archivo de log: --log-file-name

    $ ros2 run ... --ros-args --log-file-name <ruta_nombre_archivo>


Parámetros: -p o --param

    $ ros2 run ... --ros-args -p <nombre_parametro>:=<valor>

    $ ros2 run ... --ros-args  --params-file <nombre_archivo>

ros2launch

Herramienta que ayuda a describir un sistema y automatiza la ejecución en un solo comando:

    $ ros2 launch <nombre_paquete> <nombre_archivo_launch>


  • Tareas más comunes:

    • Crear nodos (y de forma sincronizada)
    • Cargar argumentos y parámetros
    • Ejecutar comandos
    • Monitorear el estado
    • Reutilizar archivos

  • Un sistema ROS 2 puede estar compuesto de muchos nodos que se ejecutan en diversos procesos (e incluso en distintos dispositivos)

  • Si bien es posible ejecutar manualmente cada uno de estos nodos, se vuelve engorroso rapidamente

  • El sistema de ‘lanzamiento’ (launch) de ROS simplifica la ejecución de sistemas complejos

  • La configuración del sistema incluye:

    • programas a ejecutar
    • dónde ejecutarlos
    • qué argumentos pasarles
    • configuraciones de ROS que facilitan la reutilización de componentes en todo el sistema

  • También se encarga de monitorear el estado de los procesos lanzados, e informar y/o reaccionar ante cambios

ros2launch: launch files

  • Formatos disponibles:

    • XML: nombre_archivo_launch.xml
    • YAML: nombre_archivo_launch.yaml
    • Python: nombre_archivo_launch.py   ⬅️


Se ubican dentro de la carpeta launch dentro del paquete

Estructura para un paquete Python

📂 paquete_ejemplo
├── 📂 launch                   ⬅️
│   ├── 📄 ejemplo_launch.py    ⬅️
│   └── ...
├── 📄 package.xml
├── 📁 paquete_ejemplo
├── 📁 resource
├── 📄 setup.cfg
├── 📄 setup.py
└── 📁 test
  • Todo lo anterior se especifica en un launch file
  • Los archivos se ubican en la carpeta launch del paquete
  • El nombre debe finalizar en launch.py, launch.xml o launch.yaml
  • Pueden ser escritos en Python, XML (ROS 1) o YAML
  • Para mayor flexibilidad, se prefiere en Python porque así se tiene acceso a todas las funciones del lenguaje y las librerías (ros2/launch y ros2/launch_ros)
  • Un archivo escrito en Python puede ser más complejo y extenso que uno en XML o YAML

Archivos launch en Python

El objeto LaunchDescription se compone de una colección de “acciones”

  • Algunos ejemplos:

    • Ejecutar un nodo (launch_ros.actions.Node)
    • Ejecutar un comando (launch.actions.ExecuteProcess)
    • Agregar argumentos (launch.actions.DeclareLaunchArgument)
    • Encadenar ejecución (ej.: launch.event_handlers.OnExecutionComplete)
    • Utilizar variables de entorno (launch.substitutions.EnvironmentVariable)
    • Condiciones (launch.conditions.IfCondition)
    • Incluir otros archivos launch (launch.actions.IncludeLaunchDescription)
  • Es la descripción de un sistema
  • La descripción se expresa como una colección (lista) de entidades que representan “intenciones” del sistema (acciones) (capturan las intenciones del usuario, arquitecto)

Archivos launch en Python

Dos formas de construir el LaunchDescription

  1. Directamente en el constructor
def generate_launch_description():
    return LaunchDescription([
        <accion1>(<argumentos_accion>),
        <accion2>(<argumentos_accion>),
        # ..
    ])
  1. Añadiendo al objeto con el método add_action
def generate_launch_description():
    accion1 = <accion1>(<argumentos_accion>)
    accion2 = <accion2>(<argumentos_accion>)

    ld = LaunchDescription(<argumentos_launch>)
    ld.add_action(accion1)
    ld.add_action(accion2)
    return ld
  • Puede que la 1 sirva para launch cortos y simples
  • La 2 es más “ordenada”

Archivos launch en Python

Dos librerías principales con elementos para LaunchDescription

  • launch: Librería independiente de ROS2 (Docs)

from launch import LaunchDescription


  • launch_ros: Librería específica con acciones de ROS2 (Docs)

from launch_ros.actions import Node

La librería launch_ros son módulos específicos con clases de ROS

Archivos launch en Python

Actions: Ejecutar un nodo

    from launch_ros.actions import Node
def generate_launch_description():
    return LaunchDescription([
        Node(
            package = '<nombre_paquete>',
            executable = '<nombre_ejecutable>',
            name = '<nombre_nodo>',
            namespace = '<nombre_namespace>',
            parameters = [{                     # Pueden ser archivos
              '<nombre_parametro>': <valor>,
              # ..
            }],
            remappings = [
              ('<nombre_topic>', '<nuevo_nombre>'),
              # ..
            ],
            output = '<screen|log|both>',
            ros_arguments = [...],              # Argumentos de ROS (nivel de log, etc)
            arguments = [...],                  # Argumentos del comando
        ),
        # ..
    ])
  • launch_ros.actions -> Acciones específicas de ROS
  • Sí o sí el executable y si es necesario package
  • Por lo general se quiere tener control en el nombre del nodo (para identificarlo en los prints de consola)
  • Utilizar namespaces únicos permite al sistema iniciar múltiples nodos iguales sin tener conflicto entre los topics o nombres
  • Utilice args cuando combine argumentos de ROS y argumentos que no lo son
  • args: requiere la bandera --ros-args para argumentos de ROS
  • ros_args para una sintaxis más limpia con solo argumentos de ROS (remapping, parámetros o log-level)

Más acciones de launch_ros: docs.ros.org

Archivos launch en Python

Actions: Ejecutar un nodo

  • Ejemplo: ejecutar talker y listener de demo_nodes_py
def generate_launch_description():
    nodo_talker = Node(
        package = 'demo_nodes_py',
        executable = 'talker',
        name = 'publicador',
    )
    node_listener = Node(
        package = 'demo_nodes_py',
        executable = 'listener',
        name = 'suscriptor',
    )
    return LaunchDescription([
        nodo_talker,
        node_listener,
    ])

Reemplaza los comandos

  •   $ ros2 run demo_nodes_py talker --ros-args -r __node:=publicador
  •   $ ros2 run demo_nodes_py listener --ros-args -r __node:=suscriptor

Archivos launch en Python

Actions: Declarar argumentos

    from launch.actions import DeclareLaunchArgument
def generate_launch_description():
    return LaunchDescription([
        DeclareLaunchArgument(
            '<nombre_argumento>',
            default_value = '<valor_por_defecto>',
            description = '<descripcion>'             # Opcional
        ),
    ])


Los argumentos se agregan al final del comando

    $ ros2 launch <nombre_paquete> <archivo_launch> <nombre_argumento>:=<valor> ..

Se pueden consultar utilizando la opción --show-args

    $ ros2 launch <nombre_paquete> <archivo_launch> --show-args
  • Pueden tener valor por defecto, una descripción (opcional)
  • Si no tiene valor por defecto el argumento se vuelve obligatorio (si no se le asigna un valor dará error)
  • Se pueden definir opciones de valores válidos (choices)
  • Cualquier argumento declarado dentro de launch.LaunchDescription se expondrá como argumento:
    • En línea de comandos al ejecutar ros2 launch
    • o en IncludeLaunchDescription (al importar)

Archivos launch en Python

Substitutions: Condición de igualdad

    from launch.substitutions import EqualsSubstitution

  • Uso: EqualsSubstitution(<argumento1>, <argumento2>)

  • Si ambos son iguales devuelve true, caso contrario false

  • Ejemplo: establecer el parámetro use_sim_time según el valor del argumento modo

def generate_launch_description():
    return LaunchDescription([
        DeclareLaunchArgument('modo', default_value = 'normal'),
        Node(
            package = 'demo_nodes_py',
            executable = 'talker',
            parameters = [{
                'use_sim_time': EqualsSubstitution(LaunchConfiguration('modo'), 'prueba')
            }],
        ),
    ])
  • Chequea si los dos argumentos son iguales y devuelve un string en “true” o “false”
  • El parámetro use_sim_time (del nodo) va a ser true solo si el argumento argumento modo es igual a “prueba”

Archivos launch en Python

Substitutions: Expresión de Python

    from launch.substitutions import PythonExpression

  • Uso: PythonExpression('<expresion_evaluable_de_python>')

  • Ejemplo: ejecutar talker o listener según el valor del argumento modo

def generate_launch_description():
    modo = LaunchConfiguration('modo')

    return LaunchDescription([
        DeclareLaunchArgument('modo', default_value = 'publicador'),
        Node(
            package = 'demo_nodes_py',
            executable = PythonExpression([
                "'talker' if '", modo, "' == 'publicador' else 'listener'"
            ])
        ),
    ])
  • Si EqualsSubstitution se queda corto, podemos usar expresiones de Python para construir valores más complejos
  • Pueden acceder a variables locales del contexto
  • Deben devolver algo que se pueda a convertir a String

Archivos launch en Python

Conditions: Condición Si <verdadero> entonces

    from launch.conditions import IfCondition

  • Uso: IfCondition(<condicion>)

  • Ejemplo: Ejecutar un nodo u otro según el valor del parámetro modo

def generate_launch_description():
    return LaunchDescription([
        DeclareLaunchArgument('modo', default_value = 'publicador'),
        Node(package = 'demo_nodes_py', executable = 'talker',
            condition = IfCondition(
                EqualsSubstitution(LaunchConfiguration('modo'), 'publicador')
            )
        ),
        Node(package = 'demo_nodes_py', executable = 'listener',
            condition = IfCondition(
                EqualsSubstitution(LaunchConfiguration('modo'), 'suscriptor')
            )
        )
    ])
  • Conditions: Mecanismos para cambiar el comportamiento del launch
  • Agregar lógica en el procedimiento de ejecución de nodos
  • Encapsula una condición “Si .. entonces” al ejecutar el launch
  • La condición es una expresión de tipo string pero debería ser posible de evaluar de forma booleana (es “true” o “false”)

Archivos launch en Python

Conditions: Condición Si <verdadero> entonces

    from launch.conditions import LaunchConfigurationEquals

  • Uso: LaunchConfigurationEquals('<nombre_argumento>', <valor>)

  • Ejemplo: Ejecutar un nodo si el valor del parámetro ejecutar_listener es true

def generate_launch_description():

        Node(package = 'demo_nodes_py', executable = 'listener',

        )
    ])

Archivos launch en Python

Events: Ejecutar una o varias acciones acción después de otra

    from launch.actions import RegisterEventHandler
    from launch.event_handlers import OnProcessExit
  • Ejemplo: ejecutar el nodo talker luego de setear un parámetro
def generate_launch_description():
    ros2_param_set = ExecuteProcess(
        cmd = ['ros2 param set', '/set_parameters_callback', 'param1', '10.0'],
        shell = True,
    )
    nodo_talker = Node(package = 'demo_nodes_py', executable = 'talker')
    return LaunchDescription([
        Node(package = 'demo_nodes_py', executable = 'set_parameters_callback'),
        ros2_param_set,
        RegisterEventHandler(event_handler = OnProcessExit(
            target_action = ros2_param_set,
            on_exit = [node_listener],
        ))
    ])
  • ROS2 launch se encarga de monitorear el estado de los procesos (iniciando, corriendo, detenidos, etc) y reaccionar ante los cambios en dicho estado
  • Estos cambios se denominan eventos y se gestionan registrando un controlador de eventos
  • Permiten definir un conjunto complejo de reglas que modifican el comportamiento del archivo de forma dinámica
  • El controlador de eventos OnProcessExit se utiliza para registrar una “acción” que se ejecuta cuando finaliza otra

Recomendaciones de diseño

  • Modularidad y reutilización

  • Usar argumentos y evitar valores hardcodeados

  • Usar niveles de jerarquía

  • Evitar lógicas complejas

  1. Dividir el sistema en múltiples launch cortos y reutilizables; usar uno principal para orquestar
  2. Evitar valores hardcodeados; usar argumentos (LaunchConfiguration) para adaptar el comportamiento.
  3. Los launch de más alto nivel deben ser cortos, claros y fáciles de modificar
  4. Mantener un esquema uniforme para nodos, topics y namespaces
  5. La lógica compleja debe ir en los nodos
