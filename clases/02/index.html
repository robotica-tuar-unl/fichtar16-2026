Robot Operating System (2)

Es un conjunto de librerías de software y herramientas que ayudan a la creación de aplicaciones robóticas (SDK)

Proporciona los componentes básicos necesarios para alcanzar los objetivos más rápidamente

www.ros.org
  • ROS2 no como un sistema operativo tradicional como Windows o macOS, sino como un “traductor y kit de herramientas universal” para robots
  • Colección de herramientas y bibliotecas de software que te ayudan a crear aplicaciones para robots sin tener que reinventar la rueda

Más información: www.ros.org/blog/ecosystem

Robot Operating System (2)

Problemas que resuelve

El problema de La torre de babel

  • Problema de comunicación entre distintos elementos (sensores, actuadores, algoritmos, etc.)
  • ROS2 proporciona una forma estándar para que se comuniquen entre sí.

Ejemplo:

  • una cámara (fabricada por la empresa A), un motor (de la empresa B) y un algoritmo (el código nuestro)
  • todos hablan “lenguajes” diferentes -> ROS 2 actúa como nexo de unión
  • proporcionando una forma estándar para que estas partes se comuniquen entre sí.

Robot Operating System (2)

Problemas que resuelve

El problema de “Reinventar la rueda”

  • ROS2 proporciona paquetes para tareas comunes
  • Permite focalizar el desarrollo en los objetivos específicos del proyecto
  • No debería ser necesario escribir código propio para calcular el giro de una rueda o el mapeo de una habitación por parte de un escáner láser
  • ROS 2 proporciona “paquetes” (código preescrito) para estas tareas comunes

Robot Operating System (2)

Problemas que resuelve

El problema del “caos del hardware”

  • ROS2 abstrae el hardware generando capas
  • Las aplicaciones de alto nivel están separadas de las de bajo nivel
  • Ejemplo: se desarrolla el código para un robot de 2 ruedas, y luego se quiere cambiar a un robot de 4 ruedas
  • ROS 2 abstrae el hardware, lo que significa que tu código de alto nivel, el “cerebro”, permanece prácticamente igual aunque cambie el “cuerpo”

ROS 2 Graph

Nodos

Programa único e independiente que realiza una tarea específica

Un sistema robótico completo se compone de múltiples nodos

Ejemplo: robot aspiradora

  • Nodo A: sensor de obstáculos
  • Nodo B: seguir el recorrido
  • Nodo C: controlar las ruedas
  • Programas ejecutables de propósito único (no es 100% estricto)
  • En vez de 1 programa principal gigante que controla todo, ROS2 utiliza pequeños nodos
  • Beneficio -> Si un nodo (por ejemplo el de la cámara) falla, el resto puede seguir funcionando o pasar a un estado seguro (ejemplo, frenar un vehículo)

ROS 2 Graph

Nodos

Se compila, ejecuta y gestiona de forma individual

Se organiza en paquetes y ejecutables que pueden contener más de un nodo

Un nodo se comunica con otros nodos, que pueden estar dentro del mismo proceso, en otro o incluso en otro dispositivo

  • Se compilan, ejecutan y gestionan de forma individual
  • Se organiza en paquetes y ejecutables, cada ejecutable puede contener más de un nodo
  • Un nodo se comunica con otros nodos, que pueden estar dentro del mismo proceso, en otro o incluso en otro dispositivo

ROS 2 Graph

Topics

Transmisión de datos continua en una única dirección entre nodos

  • Analogía: estación de radio
    • 1 publicador: productores de datos
    • N suscriptores: consumidores de datos
  • Principal forma de intercambio de información entre nodos
  • Arquitectura publicador (productores de datos) / suscriptor (consumidores de datos)
  • Medio de comunicación entre nodos
  • Única dirección y transmisión continua (perfecta para datos de sensores o estado del robot)
  • Estación de radio: The node Publishes to a topic, and any node that Subscribes to it hears the message.

ROS 2 Graph

Topics

Publicador / Suscriptor

  • Al crear un publicador se le asigna un nombre de topic al cual va a publicar

  • Para suscribirse se deberá utilizar el mismo nombre

  • En un topic puede haber cero o muchos publicadores y cero o muchos suscriptores

  • Un nodo puede suscribirse a múltiples topics y publicar en múltiples topics, todo a la vez

  • Los nodos pueden publicar o subscribirse: Generalmente en un esquema 1-pub y n-subs
  • Al crear un publicador se le asigna un nombre de “topic” al cual va a publicar -> el mismo nombre va al suscriptor
  • En un topic puede haber cero o muchos publicadores y cero o muchos suscriptores
  • Un nodo puede suscribirse a múltiples nodos y publicar en múltiples nodos, todo a la vez

ROS 2 Graph

Topics

Anónimo

  • Cuando un suscriptor recibe mensajes desconoce quién los produjo
    • Ventaja: publicadores y suscrptores pueden intercambiarse sin afectar el sistema
  • Los publicadores y suscriptores pueden aparecer y desaparecer de forma dinámica
    • Desventaja: hace falta inspeccionar y depurar en vivo

Fuertemente tipado

  • Cómo definimos los tipos?
  • Los publicadores y suscriptores pueden aparecer y desaparecer según sea necesario, lo que significa que la depuración y la introspección son extensiones naturales del sistema
  • La comunicación es anónima -> cuando un suscriptor recibe datos, por lo general desconoce quién los produjo -> el beneficio es que ambos (suscriptor o publicador) pueden intercambiarse sin afectar el sistema
  • Cómo se “tipa” -> con las interfaces .. lo que sigue

ROS 2 Graph

Mensajes

Mecanismo para enviar información donde no se espera una respuesta


Lenguaje de Definición de Interfaz (IDL)

  • Lenguaje simplificado
  • Permite generar código de forma automática
  • Archivos .msg con múltiples campos y/o constantes
  • Mecanismo para enviar información donde no se espera una respuesta
  • Ejemplo, si un node lee el valor de temperatura de un sensor, puede publicar ese dato en la red de ROS2 usando el mensaje de tipo Temperature. Otros nodos podrían suscribirse a esos datos y recibirían mensajes de Temperature
  • Estructuras de datos simples -> Ejemplos: integer para la velocidad o string para un mensaje de estado
  • ROS2 es estricto -> si un nodo envía un integer y el otro nodo no espera uno, no se van a poder comunicar
  • ROS2 utiliza un lenguaje de descripción simplificado para describir/definir las interfaces: EL IDL (Interface Definition Language)
  • IDL permite generar el código fuente automaticamente para la interfaz en distintos lenguajes de programación

ROS 2 Graph

Archivos .msg

  • Campos: tipo + nombre

mensaje.msg
tipocampo1 campo1
tipocampo2 campo2

  • Constantes: tipo + NOMBRE=valor

mensaje.msg
tipoconstante NOMBRECONSTANTE=valorconstante


  • 2 tipos de campo
    • built-in (predefinidos)
    • Otro mensaje (mensaje que contiene otro mensaje)
  • Archivos .msg de texto que definen/describen los “campos” de un mensaje de ROS2.
  • Se componen de: campos y constantes
  • Campos:
    • Consiste en un tipo y un nombre separados por un espacio
    • Los nombres tienen que ser en minúscula, caracteres alfanuméricos con guión bajo para separar palabras
    • Tienen que empezar con alfanumérico y no pueden terminar en guión bajo o tener doble guión bajo
  • Constante: campos con valores por defecto
    • nunca pueden cambiar de valor
    • el nombre es todo mayúscula
    • la asignación del valor se hace con el signo ‘=’

ROS Concept

ROS 2 Graph

  • El grafo de nodos es un “mapa” de todas los nodos y las lineas que los conectan
  • Representación visual
  • la información fluye a traves de lineas específicas (“topics”) de un nodo a otro.
  • Está diseñado para que los nodos no necesiten saber de donde viene la información

ROS 2 Graph

Nombres y namespace

Nombres jerárquicos que identifican todos los elementos de ROS


Permiten

  • Organizar sistemas complejos
  • Evitar conflictos de nombres
  • Encapsulamiento (aislar componentes)
  • Reutilización de código
  • Organización jerárquica: Permiten estructurar sistemas grandes en módulos -> Evitan conflictos de nombres
  • Cada nodo trabaja dentro de su namespace. Puede acceder a: Su propio namespace / Namespaces superiores
  • Evita usar recursos incorrectos por accidente
  • Reutilización de código -> Los nodos no necesitan saber dónde están ubicados

Intro línea de comandos

Controles más comunes

  • Enter: ejecutar el comando
  • Ctrl-C: cerrar programas en ejecución
  • Shift + Ctrl-C: copiar contenido seleccionador
  • Shift + Ctrl-V: pegar del portapapeles
  • Tab: autocompletar
  • Ctrl+C por lo general se utiliza para cerrar programas ejecutandose en la terminal
  • Ctrl+Shift+C Copiar / Ctrl+Shift+V Pegar
  • La tecla Tab se puede utilizar para autocompletar (no siempre): nombre comando, carpeta, archivo o incluso opciones y valores
    • Una vez -> Autocompleta en caso que solo exista una opción (o parcialmente si hay coincidencia entre opciones)
    • Dos veces -> Muestra las opciones
