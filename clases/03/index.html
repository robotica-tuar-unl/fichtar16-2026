ROS 2 Graph

Servicios

Intercambio de datos bidireccional entre nodos bajo demanda

  • Analogía: búsqueda en la web
    • N clientes: ingresan los términos de búsqueda
    • 1 servidor: responde con los mejores resultados
  • Otro método de comunicación para nodos pero bidireccional
  • Basados en un modelo de “llamada-respuesta” (call-and-response)
  • No son continuos, solo proveen información cuando son invocados

ROS 2 Graph

Servicios

Cliente / Servidor

  • Un servicio está formado por dos partes: el cliente de servicio y el servidor de servicio
  • Solo puede existir 1 servidor por cada servicio
  • Pueden existir una cantidad N de clientes utilizando servicio

Atómicos

  • Nunca deben ser usados para procesos largos

Identificación de servicio

  • Los servicios se identifican por nombre (misma convención que topics)

  • Un servicio se refiere a la llamada a un procedimiento remoto

  • Un nodo puede realizar una llamada a un procedimiento remoto a otro nodo que realiza una acción/computo y devuelve un resultado

  • Un servicio está formado por dos partes: El servidor de servicio y el cliente de servicio

  • El servidor es el que acepta el requerimiento, realiza alguna acción/computo al respecto y devuelve un resultado

  • El cliente del servicio es el que solicita al servidor para que realice una acción/computo

  • Solo puede existir un servidor por cada nombre de servicio

  • Pueden existir una cantidad N de clientes utilizando el mismo nombre de servicio

  • Se espera que un servicio devuelva el resultado rapidamente, dado que el nodo está esperando el resultado -> Nunca deben ser usados para procesos largos

ROS 2 Graph

Servicios

Definiciones de servicio

  • Definidos en archivos .srv utilizando IDL
  • 2 partes separadas por “---”
    • Un mensaje de requerimiento
    • Un mensaje de respuesta

Cualquiera dos .msg concatenados es una descripción de servicio válida

servicio.srv
tipodecampo1 requerimiento
---
tipodecampo2 respuesta
  • Los servicios se identifican por nombre y utilizan la misma convención de nombres (pero en otro namespace)
  • Definidos en archivos .srv
  • Esta estructura se refleja en la definición de mensaje de servicio

ROS 2 Graph

Acciones

Intercambio bidireccional bajo demanda con ejecución prolongada y feedback continuo

  • Analogía: encomienda con seguimiento
    • N clientes: solicitan y siguen un pedido
    • 1 servidor: muestra la ubicación y realiza la entrega
  • Otro método de comunicación para nodos bidireccional pero de larga duración
  • Feedback y posibilidad de cancelar
  • No son continuos, solo proveen información cuando son invocados

ROS 2 Graph

Acciones

Topics + Servicios

  • Una acción está formada por dos partes: el cliente de acción y el servidor de acción
  • Solo puede existir un servidor por cada acción
  • Pueden existir una cantidad N de clientes utilizando la misma acción

Procesos de larga duración

  • Mecanismos para obtener retroalimentación (feedback) y cancelar la operación

Identificación de acciones

  • Las acciones se identifican por nombre (misma convención que topics)

  • Una accion se refiere a una llamada de procedimiento remoto de larga ejecución con retroalimentación y la habilidad de cancelar el objetivo

  • Ejemplo: la máquina de estado de alto nivel de un robot le dice al subsistema de navegación que se dirija a un punto -> esto le tomará segundos/minutos -> a lo largo del trayecto, el subsistema de navegación puede proveer feedback de que tan lejos está y la maquina de estado tiene la opción de cancelar o adelantar el trayecto al punto

  • Una acción está formada por dos partes: el cliente de acción y el servidor de acción

  • El servidor es el que acepta el requerimiento, lleva a cabo la acción, da feedback y devuelve un resultado

  • El cliente de la acción es el que solicita al servidor para que realice la tarea, recibe el feedback y puede cancelar antes de tiempo

  • Solo puede existir un servidor por cada nombre de servicio

  • Pueden existir una cantidad N de clientes utilizando el mismo nombre de servicio

  • Se espera que la ejecución sea larga -> existen mecanismos para iniciar y monitorear la conexión

ROS 2 Graph

Acciones

Definiciones de acciones

  • Definidos en archivos .action utilizando IDL
  • 3 partes separadas por “---”
    • Un mensaje de requerimiento
    • Un mensaje de respuesta
    • Un mensaje de retroalimentación (feedback)

Cualquiera tres .msg concatenados es una descripción de acción válida

accion.action
tipodecampo1 requerimiento
---
tipodecampo2 respuesta
---
tipodecampo3 retroalimentacion
  • Las acciones se identifican por nombre y utilizan la misma convención de nombres (pero en otro namespace)
  • Definidos en archivos .action
  • Esta estructura se refleja en la definición de mensaje de servicio

  • Puede haber una cantidad arbitraria de campos (o 0) de request, una cantidad arbitraria (o 0) de respuesta, una antidad arbitraria (o 0) de feedback

  • El <request_type>, <response_type>, y <feedback_type> siguen todas las mismas reglas de <type> para un mensaje

  • El <request_fieldname>, <response_fieldname>, y <feedback_fieldname> sigue todas las mismas reglas que el <fieldname> para un mensaje

Documentación: ROS Action Interface

  • El servidor será la entidad que reciba un mensaje de 2 enteros juntos, a y b, los sume y devuelva el resultado, sum

Parámetros

Valores de configuración del nodo, que pueden variar al inicio (o durante su ejecución) sin cambiar el código

Definiciones de parámetros

  • Cada uno consiste en:
    • Una key (nombre)
    • Un valor
    • Un tipo (opcionalmente estático)
    • Un descriptor 
Tipos de parámetros:

  - `bool`
  - `int64`
  - `float64`
  - `string`
  - `byte[]`
  - `bool[]`
  - `int64[]`
  - `float64[]`
  - `string[]`

  • Se utilizan para configurar nodos al inicio (o durante su ejecución) sin cambiar el código

  • Se identifican por nombre del nodo, namespace del nodo, nombre del parámetro y namespace del parámetro (misma convención que topics)

  • Cada parámetro consiste en una key (nombre), un valor y una descripción

  • La key es una string

  • el valor puede ser bool, int64, float64, string, byte[], bool[], int64[], float64[] or string[]

  • El descriptor por defecto está vacío pero puede contener descripción, rango de valores, información del tipo y restricciones adicionales

  • Cada parámetro tiene un tipo pre-definido -> intentar cambiarlo falla en runtime, para preever errores comunes (poner un valor de bool en un int)

  • Esto se puede cambiar en el descriptor del parámetro con la variable dynamic_typing en True

ROS 2 Graph

Parámetros

El nodo debe declarar todos los parámetros que aceptará (opcional)


Identificación de parámetros

  • Se identifican por: (misma convención que topics)
    • nombre del nodo ➡️ namespace del nodo ➡️ nombre del parámetro ➡️ namespace del parámetro
  • Están atados al nodo: cuando este termina su ejecución se pierde su valor

  • Por defecto, un nodo necesita declarar todos los parámetros que aceptará durante su tiempo de vida

  • El tipo y el nombre deben estar bien definidos al tiempo de inicio del nodo -> reduce los errores de configuración posteriores

  • Esto se puede cambiar con el argumento allow_undeclared_parameters en True -> permite setear parámetros no declarados

  • Los parámetros están asociados a nodos individuales

  • El tiempo de vida está atado al tiempo de vida del nodo -> cuando este termina su ejecución se pierden (excepto que se implemente un método de recarga al inicio)

Parámetros

Las operaciones con los parámetros utilizan servicios creados por defecto por ROS2

  • Para facilitar la llamadas a estos servicios se creó el comando ros2 param


  • El valor inicial se puede setear mediante linea de comando o a través de archivos de configuración (YAML)

  • Las operaciones con los parámetros utilizan servicios creados por defecto cuando se inicia el nodo

  • Para evitar realizar llamadas a estos servicios de forma manual se creó el comando ros2 param para facilitar la interacción con los parámetros de los nodos

  • El valor inicial se puede setear mediante linea de comando o a través de archivos de configuración YAML

  • Los nodos pueden reaccionar ante cambios de los parámetros en 3 momentos distintos:

    • Antes de setear los parámetros (pero sin modificar la lista ni los valores) -> ej: un parámetro2 se necesita cambiar cuando cambia parametro1
    • Al setear los parámetros: permite inspeccionar los valores y rechazar ciertos cambios
    • Luego de setear los parámetros (comportamiento luego de que los cambios fueron aceptados satisfactoriamente)
